## 4. Root Cause Analysis

### Fault A — Wrong tunnel mode on Tunnel2

The student configured `tunnel mode ipsec ipv4` (VTI mode) on Tunnel2 and also applied `tunnel protection ipsec profile`. These two patterns are **mutually exclusive**:

```
┌──────────────────────────────────────────────────────────┐
│  tunnel mode ipsec ipv4                                  │
│  ├─ Creates a pure IPsec VTI                             │
│  ├─ Registers ONE protection entry for the peer          │
│  └─ No GRE header — multicast (OSPF) cannot traverse     │
│                                                          │
│  tunnel mode gre ip  +  tunnel protection ipsec profile  │
│  ├─ Creates a GRE tunnel wrapped in IPsec                │
│  ├─ GRE header carries multicast (OSPF hellos pass)      │
│  └─ IPsec encrypts the GRE frame, not the raw IP packet  │
└──────────────────────────────────────────────────────────┘
```

Tunnel1 is already a VTI (`tunnel mode ipsec ipv4`) to the same peer (`4.4.4.4 ↔ 1.1.1.1`). When Tunnel2 also tried to register as a VTI to the same peer, R4 could not register the second protection entry — producing `linestate protection reg down`.

**ENCOR Exam Objective:** 2.2.b — GRE and IPsec tunneling

---

### Fault B — OSPF router-id misconception

Not a configuration fault. Cisco IOS requires router-id uniqueness **within** a single OSPF process (domain), not across processes on the same router. Using `4.4.4.4` as the router-id for OSPF 1, OSPF 2, and OSPF 3 on R4 is correct and is exactly what the solution config specifies.

---

### Fault C — Loopback3 missing on R4

The OSPF 3 `network 10.4.4.6 0.0.0.0 area 0` statement was configured, but `interface Loopback3` was never created. OSPF does not generate an LSA for a network statement that matches no active interface — the adjacency forms correctly, but there is nothing to advertise for `10.4.4.6`.

---

### Fault D — OSPF 2 router-id instability (follow-up)

R1 OSPF 2 auto-selected `172.16.14.1` (the Tunnel0 IP) as its router-id because no explicit `router-id` was configured. When Tunnel2 was reconfigured, interface churn caused OSPF 2 to drop and it never reformed. The fix is to anchor OSPF 2 to `Loopback0` on both routers.

## 7. Lessons Learned

### Root Cause Categories
- **Fault A:** Tunnel mode misconfiguration — wrong mode applied to GRE-over-IPsec tunnel
- **Fault C:** Missing interface — OSPF network statement configured before the interface existed

---

### ENCOR Exam Traps

**`tunnel mode ipsec ipv4` vs `tunnel mode gre ip` + `tunnel protection`**

These are the two GRE/IPsec patterns tested on the exam. The key discriminator is **multicast**:

| Need OSPF/EIGRP over the tunnel? | Use this pattern |
|----------------------------------|-----------------|
| No — static routes only | `tunnel mode ipsec ipv4` (VTI) |
| Yes — dynamic routing required | `tunnel mode gre ip` + `tunnel protection ipsec profile` |

The exam will describe a GRE-over-IPsec tunnel that won't form OSPF neighbors. The answer is always the wrong tunnel mode (`ipsec ipv4` instead of `gre ip`).

**OSPF router-id uniqueness scope**

Router-ids must be unique per OSPF *process*, not per router. Reusing `4.4.4.4` across OSPF 1, 2, and 3 on the same router is valid. Always set it explicitly with `router-id <loopback-ip>` — never let IOS auto-select from a tunnel interface.

---

### Preventive Checklist

When configuring GRE-over-IPsec:

- [ ] `show interface TunnelX` — confirm `Tunnel protocol/transport GRE/IP`, not `IPSEC/IP`
- [ ] `show interface TunnelX` — confirm `Tunnel protection via IPSec (profile "...")` is present
- [ ] Create all test-prefix loopbacks **before** adding `network` statements in OSPF
- [ ] Set `router-id` explicitly in every OSPF process — never rely on auto-selection
- [ ] When two tunnels share the same IPsec profile and peer, add `shared` to `tunnel protection ipsec profile <name> shared`
