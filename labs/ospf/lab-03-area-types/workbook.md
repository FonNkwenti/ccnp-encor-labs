## 1. Concepts & Skills Covered

**Exam Objective:** 3.2.a — OSPF area types: stub, totally stubby, NSSA

OSPF area types are one of the most heavily tested topics on the ENCOR exam — not just which commands to use, but why the area type exists and what it does to the LSDB. This lab puts Area 1 through the Totally Stubby treatment and Area 2 through NSSA configuration, with an external redistribution scenario that mirrors real enterprise edge designs. You will also bring up a new router (R6) and verify that stub area semantics appear correctly in every router's routing table.

---

### The LSA Taxonomy

OSPF's behaviour in an area depends entirely on which LSA types are permitted to enter it. You cannot understand area types without understanding LSAs first.

| LSA Type | Name | Scope | Generated By | Carries |
|----------|------|-------|--------------|---------|
| 1 | Router LSA | Area-local | Every router | Directly connected links |
| 2 | Network LSA | Area-local | DR (broadcast/NBMA only) | Multi-access segment details |
| 3 | Summary LSA | Area-to-area | ABR | Inter-area prefixes (routes between areas) |
| 4 | ASBR Summary LSA | Area-to-area | ABR | Location of an ASBR |
| 5 | AS External LSA | Entire OSPF domain | ASBR | External routes (redistributed) |
| 7 | NSSA External LSA | NSSA area only | ASBR inside NSSA | External routes inside a stub-like area |

Types 1 and 2 are always present — they are the core link-state database. Types 3, 4, 5, and 7 are what area types restrict.

---

### Regular vs. Stub vs. Totally Stubby vs. NSSA

| Area Type | Type 3 (inter-area) | Type 4/5 (external) | Type 7 (NSSA external) | Default route injected |
|-----------|--------------------|--------------------|----------------------|------------------------|
| Regular | Yes | Yes | No | No (unless configured) |
| Stub | Yes | No | No | Yes (by ABR) |
| Totally Stubby | No (except default) | No | No | Yes (by ABR) |
| NSSA | Yes | No | Yes (locally generated) | Optional |
| Totally Stubby NSSA | No (except default) | No | Yes (locally generated) | Yes (by ABR) |

**Stub area:** Blocks Type 5 external LSAs. Allows Type 3 inter-area LSAs. Useful when the area has no ASBR and you want to reduce the LSDB size while still knowing all inter-area routes.

**Totally Stubby area (Cisco extension):** Blocks Type 3, 4, and 5 LSAs — all except the single default-route Type 3 injected by the ABR. The smallest possible LSDB. Routers in the area use a single default to reach everything outside; only the ABR needs the full routing table.

**NSSA (Not-So-Stubby Area):** Blocks Type 5 externals from entering, but allows an ASBR inside the area to originate Type 7 LSAs. The ABR translates Type 7 to Type 5 when flooding into Area 0. This lets you attach an ASBR in an area that otherwise cannot have one.

---

### ABR vs. Internal Router Configuration

This distinction is a common exam trap:

| Command | Where it goes | Effect |
|---------|--------------|--------|
| `area 1 stub no-summary` | **ABR only** | Makes area Totally Stubby — suppresses Type 3 LSAs into the stub, injects a single default |
| `area 1 stub` | ABR and all internal routers | Required on every router in the area; must match on all neighbours |
| `area 2 nssa` | ABR and all internal routers | Enables NSSA on this router for area 2 |

If an internal router omits `area X stub`, OSPF will detect a stub-area mismatch in Hello packets and refuse to bring up the adjacency.

---

### Type 7 LSA and the ABR Translation Rule

When an ASBR inside an NSSA redistributes an external route, it originates a **Type 7 LSA**. Type 7 LSAs are flooded only within the NSSA — they never cross the ABR into Area 0.

The ABR translates each Type 7 LSA into a **Type 5 LSA** and floods it into Area 0. IOS translation rules:

- Translation is performed by the **highest-router-ID ABR** if multiple ABRs border the NSSA.
- The translated Type 5 carries the ASBR's router-ID as the forwarding address.
- If the NSSA ABR also has `no-summary` configured, the area becomes a Totally Stubby NSSA (no Type 3 LSAs either, but Type 7 still originates locally).

---

### Scoped Redistribution into NSSA

When an ASBR uses `redistribute connected`, IOS redistributes **all** connected interfaces — including transit links and LAN segments that are already in OSPF via `network` statements. This creates duplicate entries: the same subnet appears as both an OSPF intra-area route (O) and a Type 7 external route (O N2), which can cause suboptimal routing.

The solution: scope redistribution with a prefix-list and route-map.

```
ip prefix-list EXTERNAL_LOOPS seq 5 permit 172.16.5.0/24
ip prefix-list EXTERNAL_LOOPS seq 10 permit 172.16.6.0/24

route-map REDIST_EXT permit 10
 match ip address prefix-list EXTERNAL_LOOPS

router ospf 1
 redistribute connected subnets route-map REDIST_EXT
```

Only prefixes matched by the route-map are redistributed. Transit and LAN subnets already participating in OSPF are excluded.

## 2. Topology & Scenario

**Scenario:** MegaCorp's network team has completed the multi-area OSPFv2/OSPFv3 build and the DR/BDR tuning from the previous lab. Two new requirements have arrived:

1. **Area 1 (branch offices)** must be optimised. The branch routers (R4, R6) have limited memory and need the smallest possible LSDB — they should only carry intra-area routes and a single default route to reach everything else. A new branch router, R6, is also being added to Area 1.

2. **Area 2 (edge site)** has a router (R5) that connects to two simulated ISP circuits. R5 needs to redistribute those external prefixes into OSPF so the rest of the network can reach them. Because Area 2 cannot receive Type 5 LSAs from outside, NSSA is required.

Your job is to bring R6 online in Area 1, convert Area 1 to Totally Stubby, and configure Area 2 as an NSSA with external redistribution on R5.

## 5. Lab Challenge: Core Implementation

### Task 1: Bring R6 Online in Area 1

- On R2, add and configure the new transit interface toward R6 using the 10.1.26.0/30 subnet. Configure OSPF point-to-point network type on this interface and assign it to Area 1. Apply equivalent OSPFv3 configuration.

### Task 2: Convert Area 1 to a Totally Stubby Area

- On R2 (ABR): configure Area 1 with the ABR-specific stub option that also suppresses inter-area Type 3 summary LSAs. This makes Area 1 Totally Stubby. Apply the equivalent OSPFv3 configuration.

**Verification:** `show ip route ospf` on R4 must show only intra-area routes (code `O`) plus a single default route (code `O*IA`). No `O IA` routes for Area 0 or Area 2 prefixes should appear. Run the same check on R6.

---

### Task 3: Configure Area 2 as NSSA and Redistribute External Routes

- On R5, add two new loopback interfaces: Loopback1 with 172.16.5.1/24 (IPv6: 2001:DB8:172:5::1/64) and Loopback2 with 172.16.6.1/24 (IPv6: 2001:DB8:172:6::1/64). These simulate connected ISP circuits. Do NOT add these interfaces to OSPF directly.
- On R5, create prefix-lists that match only those two external prefixes — not the transit links or LAN that are already in OSPF. Create route-maps referencing those prefix-lists, one for IPv4 and one for IPv6.
- On R3 (ABR): configure Area 2 as NSSA. Apply the equivalent OSPFv3 configuration.
- On R5 (ASBR): configure Area 2 as NSSA. Redistribute the external loopbacks into OSPF using the scoped route-maps created above. Apply the equivalent OSPFv3 configuration.

**Verification:** `show ip ospf database nssa-external` on R5 must show two Type 7 LSAs for 172.16.5.0 and 172.16.6.0, both with advertising router 5.5.5.5. `show ip route ospf` on R1 must show `O E2` routes for both 172.16.5.0/24 and 172.16.6.0/24 via R3. `show ip route ospf` on R4 must show those same two external routes (R4 learns them as Type 5 externals forwarded through Area 1 — but wait, Area 1 is Totally Stubby, so R4 should NOT see them — only the default route). ## 7. Verification Cheatsheet

### OSPF Area Type Configuration

```
! ABR — Totally Stubby (suppresses all Type 3 except default):
router ospf 1
 area <id> stub no-summary

! ABR — NSSA:
router ospf 1
 area <id> nssa

! Internal router — Stub or Totally Stubby:
router ospf 1
 area <id> stub

! Internal ASBR — NSSA with redistribution:
router ospf 1
 area <id> nssa
 redistribute connected subnets route-map <MAP>

! OSPFv3 — area types go inside address-family:
router ospfv3 1
 address-family ipv6 unicast
  area <id> stub no-summary
  area <id> nssa
 exit-address-family
```

| Command | Purpose |
|---------|---------|
| `area X stub no-summary` | ABR: Totally Stubby (no Type 3 except default) |
| `area X stub` | Internal router: accept stub area semantics |
| `area X nssa` | Enable NSSA on ABR or internal router |
| `area X nssa no-summary` | ABR: Totally Stubby NSSA (no Type 3, but Type 7 still works) |

> **Exam tip:** `no-summary` on an ABR affects Type 3 LSA flooding direction (into the area). It does NOT affect Type 7 generation — an NSSA ASBR can still originate Type 7 regardless of whether the ABR has `no-summary`.

### Redistribution with Prefix-List Scoping

```
ip prefix-list <NAME> seq <N> permit <prefix/len>

route-map <MAP> permit 10
 match ip address prefix-list <NAME>

router ospf 1
 redistribute connected subnets route-map <MAP>
```

| Command | Purpose |
|---------|---------|
| `ip prefix-list` | Identify specific prefixes to include or exclude |
| `route-map ... permit` | Permit matched prefixes through |
| `redistribute connected subnets route-map` | Redistribute only what the route-map permits |
| `ipv6 prefix-list` | IPv6 equivalent for OSPFv3 redistribution |

> **Exam tip:** Always scope redistribution into OSPF. Bare `redistribute connected subnets` will inject ALL connected subnets, including OSPF transit links, creating both internal and external LSAs for the same prefix.

### Verification Commands

| Command | What to Look For |
|---------|-----------------|
| `show ip ospf neighbor` | All expected neighbors in FULL state |
| `show ip ospf` | Area type shown as "stub area" or "not-so-stubby" |
| `show ip route ospf` | R4/R6: only `O` + `O*IA` default; no `O IA`, `O E1`, `O E2` |
| `show ip ospf database` | Confirm LSA types present per area |
| `show ip ospf database nssa-external` | Type 7 LSAs on R5/R3 with correct advertising router |
| `show ip ospf database external` | Type 5 LSAs on R3 (translated from Type 7) |
| `show ospfv3 neighbor` | IPv6 adjacencies in FULL state |
| `show ipv6 route ospf` | IPv6 routing table mirrors IPv4 area type behaviour |
| `ping 172.16.5.1 source loopback0` | End-to-end reachability to NSSA external from R1 |

### LSA Type Quick Reference

| LSA Type | Code in LSDB | Present in Stub? | Present in Totally Stubby? | Present in NSSA? |
|----------|-------------|-------------------|---------------------------|-----------------|
| 1 (Router) | `router` | Yes | Yes | Yes |
| 2 (Network) | `network` | Yes | Yes | Yes |
| 3 (Summary) | `summary` | Yes (all) | Only default (0.0.0.0/0) | Yes |
| 4 (ASBR Summary) | `asbr-summary` | No | No | No |
| 5 (External) | `external` | No | No | No |
| 7 (NSSA External) | `nssa-external` | No | No | Yes (locally generated) |

### Common OSPF Area Type Failure Causes

| Symptom | Likely Cause |
|---------|-------------|
| Adjacency stuck in EXSTART/INIT | Stub area mismatch — one side has `area X stub`, other does not |
| R4/R6 shows `O IA` routes in stub | ABR missing `no-summary` (regular stub, not totally stubby) |
| No external routes on R1 despite R5 redistributing | R3 missing `area 2 nssa` — Type 7 not being translated |
| All connected subnets redistributed (duplicates) | Missing route-map scope on `redistribute connected` |
| NSSA Type 7 visible on R5 but not on R1 | ABR (R3) is not translating — check `show ip ospf database external` on R3 |
| R6 stuck at 2-WAY or DOWN | Missing `area 1 stub` on R6 — stub area mismatch in Hello |

### Ticket 1 — R6 Has No OSPF Neighbors

The branch team reports that a newly added router (R6) cannot reach any resources outside Area 1. Console access to R6 confirms the interfaces are up but `show ip ospf neighbor` returns no output.

**Inject:** `python3 scripts/fault-injection/inject_scenario_01.py --host <eve-ng-ip>`

**Success criteria:** R6 shows FULL adjacency with R2 and R4. `show ip route ospf` on R6 displays the expected Area 1 routes plus the default route.

<details>
<summary>Click to view Diagnosis Steps</summary>

```bash
! On R6

show ip ospf interface gi0/0       ! check area, hello/dead, network type

! On R2
show ip ospf neighbor              ! R6 should appear — check if 2-WAY or missing
show ip ospf database              ! check if stub mismatch hello rejected
show log                           ! %OSPF-5-ADJCHG or stub mismatch messages
```

Stub area mismatch causes hellos to be rejected. The router with the missing `area X stub` will not form an adjacency because the E-bit in the Hello packet will not match.
</details>

### Ticket 3 — Duplicate OSPF Routes for Area 2 Transit Subnets

A routing anomaly is reported: `show ip route ospf` on R3 shows both `O` (internal) and `O N2` (NSSA external) entries for the 10.2.35.0/30 subnet and for 192.168.2.0/24. The concern is that R5 is advertising its own transit and LAN subnets as external routes.

**Inject:** `python3 scripts/fault-injection/inject_scenario_03.py --host <eve-ng-ip>`

**Success criteria:** `show ip route ospf` on R3 shows 10.2.35.0/30 and 192.168.2.0/24 as `O` (intra-area) only. No `O N2` or `O E2` duplicates exist for those prefixes. The external routes 172.16.5.0/24 and 172.16.6.0/24 still appear correctly as `O E2` on R1.

<details>

show run | section redistribute        ! check route-map association
show run | section prefix-list         ! check if EXTERNAL_LOOPS prefix-list is correct
