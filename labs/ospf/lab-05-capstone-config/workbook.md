## 1. Concepts & Skills Covered

**Exam Objective:** 3.2.a (OSPF area types, LSA types, adjacency behavior) and 3.2.b (OSPF neighbor relationships, DR/BDR election, inter-area routing, summarization, filtering)

This capstone integrates every OSPF concept from labs 00 through 04 into a single build-from-scratch challenge. You start from a clean slate — IP addressing is pre-configured but the OSPF control plane is entirely absent. You must configure and verify multi-area OSPFv2 and OSPFv3 simultaneously, choosing the correct area types, LSA filtering mechanisms, summarization strategy, and route filtering technique for each part of the topology.

---

### Topic 1: OSPF Multi-Area Hierarchy

OSPF organizes routers into areas to control LSDB size and LSA propagation. Area 0 (the backbone) must connect all other areas. An Area Border Router (ABR) is a router with interfaces in both Area 0 and a non-backbone area — it maintains one LSDB per area and generates Type 3 (Summary) LSAs to describe inter-area prefixes.

Key LSA types and their propagation scope:

| LSA Type | Name | Scope | Generated By |
|----------|------|-------|-------------|
| Type 1 | Router LSA | Intra-area | Every OSPF router |
| Type 2 | Network LSA | Intra-area | DR on broadcast segments |
| Type 3 | Summary LSA | Inter-area | ABR |
| Type 4 | ASBR Summary | Domain-wide | ABR (about an ASBR) |
| Type 5 | AS External | Domain-wide | ASBR |
| Type 7 | NSSA External | NSSA area only | ASBR in NSSA |

Understanding which LSA types a given area type blocks is essential for both configuration and troubleshooting.

---

### Topic 2: Area Types — Stub, Totally Stubby, and NSSA

Normal areas flood all LSA types. Stub areas block Type 5 (external) LSAs. Totally stubby areas additionally block Type 3 LSAs (except the default). NSSA areas allow limited external route injection via Type 7 LSAs, which are translated to Type 5 at the ABR boundary.

**Area type comparison:**

| Area Type | Blocks Type 3? | Blocks Type 5? | Allows Redistribution? | Default Route? |
|-----------|----------------|----------------|----------------------|----------------|
| Normal | No | No | Yes | No (unless configured) |
| Stub | No | Yes | No | Yes (auto-injected) |
| Totally Stubby | Yes (except default) | Yes | No | Yes (auto-injected) |
| NSSA | No | Yes (from outside) | Yes (Type 7) | Optional |

The `area N stub no-summary` command on the ABR makes an area totally stubby — it must appear on the ABR only. The `area N stub` command must appear on all routers within the area.

For NSSA, `area N nssa` must be configured on all routers in the area. The ASBR uses `redistribute` to inject routes as Type 7 LSAs. The ABR translates the highest-priority Type 7 to Type 5 for the backbone.

---

### Topic 3: OSPF Summarization — ABR and ASBR

**Inter-area summarization (ABR):** The `area N range <prefix> <mask>` command on an ABR suppresses individual Type 3 LSAs and replaces them with a single aggregate Type 3 LSA. The `not-advertise` keyword additionally suppresses the aggregate — no LSA of any kind is generated for that prefix.

```
! On ABR R2: summarize Area 1 loopbacks into a /22
area 1 range 10.1.4.0 255.255.252.0

! Suppress a specific prefix — no Type 3 generated
area 1 range 10.1.6.0 255.255.255.0 not-advertise
```

**ASBR summarization:** The `summary-address <prefix> <mask>` command under `router ospf` on the ASBR aggregates external routes before generating Type 5 (or Type 7 in NSSA) LSAs. Individual component routes are suppressed.

For OSPFv3, the ABR equivalent uses `area N range <prefix/len>` under the address-family; the ASBR uses `summary-prefix <prefix/len>` under the address-family.

---

### Topic 4: DR/BDR Election on Broadcast Segments

On multi-access (broadcast) segments, OSPF elects a Designated Router (DR) and Backup DR (BDR) to reduce adjacency count. All routers form full adjacency only with the DR and BDR. DROTHERs are in 2-WAY state with each other.

Election criteria (in order):
1. Highest OSPF interface priority (default 1; 0 = never elected)
2. Highest router-ID (tiebreaker)

Priority is set per-interface with `ip ospf priority <0-255>`. The election is non-preemptive — a new higher-priority router joining after election will not displace the current DR unless the current DR fails.

---

### Topic 5: OSPF Route Filtering with Distribute-Lists

Unlike EIGRP, OSPF `distribute-list` does not prevent LSAs from being placed in the LSDB — it only filters routes from being installed in the local routing table. The effect is local to the router where the command is applied and does not influence other routers' routing tables.

```
! Block 10.1.5.0/24 from being installed in R1's routing table
ip prefix-list BLOCK_10_1_5 seq 5 deny 10.1.5.0/24
ip prefix-list BLOCK_10_1_5 seq 10 permit 0.0.0.0/0 le 32

router ospf 1
 distribute-list prefix BLOCK_10_1_5 in
```

The `in` keyword filters routes as they are being installed in the RIB. The LSA for 10.1.5.0/24 still exists in the LSDB on R1 — only the RIB entry is suppressed.

---

### Topic 6: OSPFv3 Dual-Stack with Address Families

OSPFv3 in IOS 15.x uses address-family mode, which supports both IPv4 and IPv6 unicast under a single OSPFv3 process. Interfaces are enabled with `ospfv3 <pid> ipv6 area <N>` per interface, and the address-family is declared under `router ospfv3`.

Area types and summarization syntax mirror OSPFv2 but are configured within `address-family ipv6 unicast`. OSPFv3 uses link-local addresses for hello packets, so link-local addresses must be configured on all participating interfaces.

## 5. Lab Challenge: Full Protocol Mastery

> This is a capstone lab. No step-by-step guidance is provided.
> Configure the complete OSPF solution from scratch — IP addressing is pre-configured; everything else is yours to build.
> All blueprint bullets for this chapter must be addressed.

**Your objectives:**

1. **Multi-area OSPF foundation** — Configure OSPF process 1 on all routers with explicit router-IDs. Advertise all appropriate networks. R1, R2, and R3 connect to the Area 0 broadcast segment; R2 is the ABR for Area 1; R3 is the ABR for Area 2.

2. **DR/BDR election control** — On the Area 0 shared segment, engineer the election so that R1 becomes the DR (priority 255), R2 becomes the BDR (priority 200), and R3 is always a DROTHER (priority 0).

3. **Area 1 — Totally Stubby** — Configure Area 1 as totally stubby. R2 must inject only a default route into Area 1. R4 and R6 must not receive Type 3 or Type 5 LSAs from outside Area 1.

4. **Area 2 — NSSA with redistribution** — Configure Area 2 as an NSSA. Configure R5 as an ASBR redistributing its two external loopbacks (172.16.5.0/24 and 172.16.6.0/24) into OSPF using a route-map scoped to a prefix-list. Passive interfaces must be set on R5's LAN-facing interface.

5. **Summarization — inter-area (ABR)** — On R2, summarize the four Area 1 loopback networks from R4 (10.1.4.0/24, 10.1.5.0/24, 10.1.6.0/24, 10.1.7.0/24) into a single /22 aggregate into Area 0. Additionally, suppress 10.1.6.0/24 entirely from being advertised to Area 0 using the not-advertise mechanism.

6. **Summarization — external (ASBR)** — On R5, summarize the two external loopbacks into a single aggregate before they are advertised as Type 7 LSAs into Area 2.

7. **Route filtering** — On R1, prevent 10.1.5.0/24 from being installed in the local routing table using a distribute-list applied inbound to the OSPF process. The LSDB entry must remain — only the RIB entry should be absent.

8. **Point-to-point network type** — Ensure all Area 1 and Area 2 transit links operate as OSPF point-to-point to eliminate DR/BDR overhead. Additionally, configure custom hello/dead timers (5s/20s) on the R2-R4 link.

9. **OSPFv3 dual-stack** — Configure OSPFv3 process 1 with IPv6 address families on all routers. Enable OSPFv3 on all interfaces that carry OSPFv2. Apply the same area types, summarization, and passive-interface settings in the OSPFv3 address family.

10. **End-to-end reachability** — Verify full IPv4 and IPv6 connectivity from PC1 to PC2, and between all router loopbacks. Confirm that the distribute-list filter on R1 correctly blocks 10.1.5.0/24 from R1's routing table while leaving all other permitted routes reachable.

## 7. Verification Cheatsheet

### OSPF Process and Neighbor Verification

```
show ip ospf neighbor
show ip ospf neighbor detail
show ip ospf interface <int>
show ip ospf interface brief
show ip ospf border-routers
```

| Command | Purpose |
|---------|---------|
| `show ip ospf neighbor` | Neighbor state, priority, DR/BDR role |
| `show ip ospf neighbor detail` | Hello/dead timers, options flags, interface details |
| `show ip ospf interface Gi0/0` | Network type, priority, DR/BDR IPs, hello/dead intervals |
| `show ip ospf border-routers` | ABR and ASBR entries in LSDB |

> **Exam tip:** OSPF neighbors stuck in INIT indicate one-way hello reception. EXSTART/EXCHANGE failures often indicate MTU mismatch (`ip ospf mtu-ignore`) or duplicate router-IDs.

show ip ospf database

### Area Type Configuration

```
! ABR side — totally stubby:
area 1 stub no-summary

! All routers in area 1:
area 1 stub

! ABR side — NSSA:
area 2 nssa

! All routers in area 2:
area 2 nssa
```

| Command | Purpose |
|---------|---------|
| `area N stub` | Enable stub on all routers in area N |
| `area N stub no-summary` | ABR only — makes area totally stubby |
| `area N nssa` | Enable NSSA on all routers in area N |
| `area N nssa no-summary` | ABR only — NSSA with only default route (no Type 3s) |

### Summarization Commands

```
! ABR inter-area summarization:
area 1 range 10.1.4.0 255.255.252.0
area 1 range 10.1.6.0 255.255.255.0 not-advertise

! ASBR external summarization (IPv4):
summary-address 172.16.0.0 255.255.0.0

! OSPFv3 ABR:
area 1 range 2001:DB8:1:4::/62

! OSPFv3 ASBR:
summary-prefix 2001:DB8:172::/48
```

| Command | Purpose |
|---------|---------|
| `area N range <prefix> <mask>` | Summarize intra-area routes into a Type 3 LSA |
| `area N range <prefix> <mask> not-advertise` | Suppress prefix — no Type 3 generated |
| `summary-address <prefix> <mask>` | Summarize external routes before Type 5/7 generation |
| `summary-prefix <prefix/len>` | OSPFv3 ASBR external summarization |

> **Exam tip:** `area range` is on the ABR; `summary-address` is on the ASBR. Mixing them up is a common configuration error that shows up as either missing summaries or unexpectedly detailed LSAs.

### Common OSPF Failure Causes

| Symptom | Likely Cause |
|---------|-------------|
| Neighbor stuck in INIT | One-way hello; ACL blocking multicast 224.0.0.5 |
| Neighbor stuck in EXSTART | MTU mismatch; duplicate router-ID |
| No adjacency after config | Area ID mismatch; authentication mismatch; hello/dead timer mismatch |
| Missing inter-area routes | ABR not properly configured; network statement missing |
| Type 5 LSAs visible in stub area | `area N stub` missing on at least one router in the area |
| No default route in stub area | `area N stub` configured but ABR not running OSPF on Area 0 interface |
| NSSA Type 7 not translated to Type 5 | No NSSA ABR; ABR has `area N nssa no-redistribution` |
| Summary not visible in Area 0 | `area range` configured but component routes not in LSDB |
| Distribute-list has no effect | Prefix-list name typo; `in` direction not specified |
| OSPFv3 not forming | `ospfv3 pid ipv6 area N` missing on interface; link-local not configured |
