## 4. Root Cause Analysis

### Primary Cause: Missing `native` Keyword

**Technical Explanation:**

On Cisco router-on-a-stick architecture, sub-interfaces represent individual VLANs. 
The native VLAN (untagged on trunks) requires special handling:

**Incorrect Configuration (before fix):**
```
interface GigabitEthernet0/0.99
 encapsulation dot1Q 99              ← Missing 'native' keyword
 ip address 192.168.99.254 255.255.255.0
```

Without `native`, R1 interprets VLAN 99 as a **tagged** VLAN and discards untagged frames.

**Frame Flow Analysis:**

```
SW1 (Distribution)              Trunk Link              R1 (Router)
┌──────────────────┐            ─────────            ┌────────────────┐
│ VLAN 99 Native   │            Frame type           │ Gi0/0.99       │
│ Sends: UNTAGGED  │ ─────────→ UNTAGGED frames     │ Expected: TAG   │
│ (no 802.1Q tag)  │            ─────────            │ Action: DROP    │
└──────────────────┘                                  └────────────────┘
```

**Correct Configuration (after fix):**
```
interface GigabitEthernet0/0.99
 encapsulation dot1Q 99 native       ← Tells R1: "accept untagged frames"
 ip address 192.168.99.254 255.255.255.0
```

Now R1 accepts untagged VLAN 99 frames because the `native` keyword informs
the encapsulation parser: "This VLAN arrives untagged on the trunk."

### Secondary Cause: IP Address Loss

When reconfiguring the encapsulation:
```
no encapsulation dot1Q 99            ← Removes sub-interface from config
encapsulation dot1Q 99 native        ← Re-adds sub-interface
```

The sub-interface is temporarily deleted and recreated, which removes all
previously configured parameters (including IP address). The `ip address` 
command must be re-applied in the same configuration block.

### Exam Relevance (CCNP ENCOR 3.1.a)

**Blueprint Coverage:**
- 3.1.a — "Describe VLANs and trunking, including ISL and 802.1Q encapsulation"
- Sub-bullet: Native VLAN handling on trunk links and router-on-a-stick

**Common Exam Traps:**

1. **Syntax confusion:** Students write `encapsulation dot1Q native 99` (native first) 
   instead of `encapsulation dot1Q 99 native` (native last). IOS rejects the first.

2. **Missing on all sub-interfaces:** Only VLAN 99 (native VLAN) needs `native`.
   Other VLANs (.10, .20, .30) do NOT use this keyword:
   ```
   interface Gi0/0.10: encapsulation dot1Q 10        ← NO 'native'
   interface Gi0/0.99: encapsulation dot1Q 99 native ← YES 'native'
   ```

3. **Silent frame drops:** Native VLAN mismatch doesn't break the trunk or show errors
   in `show interfaces` — frames are silently dropped. Debugging requires checking
   the exact encapsulation on both ends.

4. **Reconfiguration pitfall:** Many engineers reconfigure encapsulation to fix one 
   issue but forget to restore IP address after. This turns a 1-minute fix into
   a 30-minute debugging session.

## 7. Lessons Learned

### Root Cause Category

**Native VLAN Encapsulation on Router-on-a-Stick** (L3 frame processing)

### Exam Relevance

**CCNP ENCOR 3.1.a** — "Describe VLANs and trunking, including ISL and 802.1Q 
encapsulation" explicitly tests:
- Native VLAN behavior (untagged on trunks)
- Router-on-a-stick sub-interface configuration
- The `native` keyword syntax and placement

### Common Pitfalls & Prevention

#### Pitfall 1: Forgetting the `native` Keyword

**Wrong:**
```ios
interface Gi0/0.99
 encapsulation dot1Q 99        ← Missing 'native' → silent frame drops
```

**Right:**
```ios
interface Gi0/0.99
 encapsulation dot1Q 99 native  ← Tells IOS: accept untagged frames
```

**Prevention:** Always check workbook or reference config for native VLAN sub-interface.
Native VLAN is the **only** sub-interface that uses the `native` keyword.

#### Pitfall 2: Incorrect Keyword Placement

**Wrong:**
```ios
encapsulation dot1Q native 99   ← Syntax error; native must come last
```

**Right:**
```ios
encapsulation dot1Q 99 native   ← Correct: VLAN ID first, native last
```

**Prevention:** VLAN ID always comes before `native` in the syntax.

#### Pitfall 3: IP Address Loss During Reconfig

**Wrong approach:**
```ios
no encapsulation dot1Q 99
encapsulation dot1Q 99 native
! Forgot to re-add IP → sub-interface unassigned
```

**Right approach:**
```ios
no encapsulation dot1Q 99
encapsulation dot1Q 99 native
ip address 192.168.99.254 255.255.255.0  ← Always re-apply after reconfig
```

**Prevention:** When modifying encapsulation, immediately apply the `ip address` 
command in the same config block.

#### Pitfall 4: Confusing Native VLAN Across Devices

| Device | Native VLAN Behavior | Configuration |
|--------|---------------------|----------------|
| **Switch** | Automatically forwards VLAN 99 untagged; no special config needed | `switchport trunk native vlan 99` (just sets native, doesn't tag/untag) |
| **Router** | Must explicitly accept untagged frames; uses `native` keyword in encapsulation | `encapsulation dot1Q 99 native` (tells router to process untagged frames) |

**Prevention:** Remember: switches handle native VLAN automatically; routers need 
the `native` keyword to do the same.

### Design Principles

> **Principle:** On router-on-a-stick, native VLAN sub-interface configuration is 
> the inverse of how switches handle native VLAN. Switches forward VLAN N untagged 
> by default; routers must be told explicitly via the `native` keyword.

### Preventive Checklist

When configuring router-on-a-stick:

- [ ] Confirm which VLAN is native (workbook, topology diagram)
- [ ] For native VLAN sub-interface: `encapsulation dot1Q <native-vlan> native`
- [ ] For tagged VLAN sub-interfaces: `encapsulation dot1Q <vlan>` (no native)
- [ ] Apply `ip address` immediately after encapsulation in config block
- [ ] Verify with `show interfaces Gi0/0.99` — confirm native keyword is accepted
- [ ] Test with ping from both directions (R1 → switch, switch → R1)
- [ ] Check for silent frame drops if tagged/native mismatch occurs
